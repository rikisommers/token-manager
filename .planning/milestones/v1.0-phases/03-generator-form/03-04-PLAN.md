  <what-built>
Complete save/load/overwrite cycle in the generator form (Plans 03-01 through 03-03):
- POST /api/collections and PUT /api/collections/[id] API endpoints
- "Save to Database" button with name-prompt modal and duplicate-name overwrite confirmation
- "Load Collection" button with scrollable collection list and unsaved-changes guard
- "Editing: [name]" indicator, dirty flag, and Clear Form integration
  </what-built>
  <how-to-verify>
Visit http://localhost:3000/generate and run through each scenario:

**Scenario 1 — Save new collection:**
1. Add at least one token group and token value to the form.
2. Click "Save to Database".
3. In the modal, type a new collection name (e.g. "test-collection-1") and click Save.
4. Expected: Modal closes, toast "Saved to database: test-collection-1" appears and auto-dismisses. "Editing: test-collection-1" label appears near the Save/Load buttons.

**Scenario 2 — Duplicate-name overwrite confirmation:**
1. Click "Save to Database" again.
2. Type the same name "test-collection-1" and click Save.
3. Expected: Modal shows "A collection named 'test-collection-1' already exists. Overwrite it?" with Overwrite and Cancel buttons.
4. Click Overwrite. Expected: Modal closes, toast "Saved to database: test-collection-1" appears.

**Scenario 3 — Load collection:**
1. Clear the form (click "Clear Form", confirm). "Editing:" label should disappear.
2. Click "Load Collection".
3. Expected: Dialog opens with a spinner briefly, then shows "test-collection-1" in the list.
4. Click the collection name.
5. Expected: Dialog closes, form populates with the previously saved tokens, toast "Loaded: test-collection-1" appears, "Editing: test-collection-1" label reappears.

**Scenario 4 — Edit and overwrite loaded collection:**
1. With "test-collection-1" loaded, modify a token value.
2. Click "Save to Database".
3. Expected: Modal opens with "test-collection-1" pre-filled in the name field.
4. Click Save without changing the name.
5. Expected: Goes through the 409 overwrite confirmation (since the name exists), then on Overwrite — saves successfully and "Editing: test-collection-1" remains set.

**Scenario 5 — Unsaved-changes guard:**
1. Modify a token value (making the form dirty).
2. Click "Load Collection".
3. Expected: dialog opens (no guard before dialog opens — guard fires when selecting a collection).
4. Click a collection name.
5. Expected: Browser confirmation "You have unsaved changes. Loading a collection will replace them. Continue?" appears.
6. Click Cancel. Expected: Form state unchanged, dialog still open.
7. Click the collection again and confirm. Expected: Form loads the collection.

**Scenario 6 — Empty load dialog:**
1. If you have access to a fresh database with no collections: Open "Load Collection". Expected: Dialog shows "No collections saved yet."
(Skip this scenario if collections already exist — the empty state is code-verified.)

**Scenario 7 — Existing export functionality with loaded collection:**
1. Load any collection so the form has token data.
2. Click "Preview JSON". Expected: JSON preview dialog opens showing the loaded collection's tokens.
3. Click "Download JSON". Expected: JSON file downloads.
4. (If GitHub configured) Click "Push to GitHub" / "Import from GitHub". Expected: directory picker opens as normal.
  </how-to-verify>
