Modify the `authorize()` callback in `src/lib/auth/nextauth.config.ts` to:

1. **Replace `return null` with `throw new Error(...)` for meaningful failure cases.** The current code returns `null` for user-not-found and wrong-password cases — this produces the generic "CredentialsSignin" error in the client. Locked decision requires specific messages. New authorize() logic:

```typescript
async authorize(credentials) {
  if (!credentials?.email || !credentials?.password) return null; // keep null for empty fields (form validation prevents this)
  await dbConnect();
  const user = await User.findOne({ email: credentials.email.toLowerCase() });
  if (!user) throw new Error('No account found with that email');
  if (user.status === 'disabled') throw new Error('Incorrect password'); // generic — same as wrong pw, don't reveal status
  // Note: 'invited' status users who have set a password CAN sign in (they are not yet 'active' until Phase 20 sets status)
  // Only 'disabled' is explicitly blocked
  const valid = await bcrypt.compare(credentials.password, user.passwordHash);
  if (!valid) throw new Error('Incorrect password');
  return { id: user._id.toString(), email: user.email, name: user.displayName, role: user.role };
},
```

2. **Add `name: user.displayName`** to the return object. This populates `session.user.name` (part of NextAuth's default session type — no type augmentation needed). The jwt callback must also propagate `name` if not already doing so:

```typescript
async jwt({ token, user }) {
  if (user) {
    token.id   = (user as any).id;
    token.role = (user as any).role;
    token.name = (user as any).name; // add this line
  }
  if (token.email === process.env.SUPER_ADMIN_EMAIL) {
    token.role = 'Admin';
  }
  return token;
},
```

The session callback already maps `session.user.id` and `session.user.role`. `session.user.name` is handled automatically by NextAuth from `token.name` — no explicit mapping needed in the session callback.

Keep everything else in `nextauth.config.ts` unchanged (session strategy, maxAge, pages.signIn, secret).

Create `src/app/api/auth/setup/route.ts` with GET and POST handlers.

**GET handler** — used by the setup page on mount to determine whether to show the form or redirect:
- `await dbConnect()`
- `const count = await User.countDocuments()`
- If `count === 0`: return `NextResponse.json({ setupRequired: true, email: process.env.SUPER_ADMIN_EMAIL })`
  - Note: email is ONLY included when setup is actually required — never expose it once setup is complete
- If `count > 0`: return `NextResponse.json({ setupRequired: false })`

**POST handler** — creates the first Admin user:
- `await dbConnect()`
- Re-check `User.countDocuments()` — if > 0, return `NextResponse.json({ error: 'Setup already complete' }, { status: 403 })`
- Parse `{ displayName, password }` from `request.json()` — type as `{ displayName: string; password: string }`
- Read `email = process.env.SUPER_ADMIN_EMAIL` — if falsy, return `{ error: 'SUPER_ADMIN_EMAIL env var not set' }` with status 500
- Validate: `!displayName?.trim()` or `!password || password.length < 8` → return `{ error: 'Invalid input' }` with status 400
- Hash: `const passwordHash = await bcrypt.hash(password, 12)`
- Create: `await User.create({ displayName: displayName.trim(), email: email.toLowerCase(), passwordHash, role: 'Admin', status: 'active' })`
  - CRITICAL: `status: 'active'` — User schema defaults to `'invited'`; must be explicitly set so the admin can sign in immediately
- Return `NextResponse.json({ ok: true })`
