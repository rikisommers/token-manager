**On mount (useEffect with `id` dep):**
1. Fetch `GET /api/collections/${id}/themes` → set `themes`; select first theme if any
2. Fetch `GET /api/collections/${id}` → extract groups from `col.tokens` using the existing `parseGroupsFromTokens` helper. If no such helper exists, do a simple `Object.keys(col.tokens).filter(k => !k.startsWith('$')).map(key => ({ id: key, name: key, ... }))` — but check if `TokenGeneratorForm` exports a utility for this. Since groups are parsed in `TokenGeneratorForm`, the simplest approach is to keep a lightweight local parse: top-level keys of `col.tokens` that are non-`$` objects become groups. The `masterGroups` state only needs `id`, `name`, and `children` for the matrix display.

Actually: the Themes page only needs the GROUP LIST for the matrix display. Instead of replicating group parsing, fetch groups from the collection's token tree using the same logic: `Object.entries(col.tokens as Record<string, unknown>).filter(([k, v]) => !k.startsWith('$') && typeof v === 'object' && v !== null).map(([k]) => ({ id: k, name: k, tokens: [], level: 0 }))` — this gives a flat list of top-level group objects sufficient for the matrix.

For display labels use `parseGroupPath(group.name)` (already imported in TokenGroupTree).

**Handlers:**
- `handleAddTheme(name)`: POST to `/api/collections/${id}/themes` with `{ name }`. On success, append to `themes` state and select the new theme.
- `handleDeleteTheme(themeId)`: DELETE to `/api/collections/${id}/themes/${themeId}`. On success, filter from `themes` state; if deleted was selected, select first remaining or null.
- `handleStateChange(groupId, state)`: PUT to `/api/collections/${id}/themes/${selectedThemeId}` with `{ groups: { ...selectedTheme.groups, [groupId]: state } }`. On success, update `themes` state (find theme by id, merge groups).
