## Summary

Phase 12 adds theme awareness to the two existing export flows. The SD export (via `buildTokens()`) already accepts arbitrary token JSON — the only new work is (a) building a merged token object that reflects the chosen theme's group states, and (b) injecting a comment header into the output for code formats. These changes are self-contained in the Config page and `build-tokens` API route; no changes to `style-dictionary.service.ts` are needed.

The Figma Variables export requires a more substantial rework of `POST /api/export/figma`. The current implementation sends a single flat `variables` array with a hard-coded `default` mode. Multi-mode export requires restructuring the payload to use the Figma Variables API's `variableModes` and `variableModeValues` arrays — one mode per enabled theme plus the mandatory "Default" mode. The collection must already exist in Figma (same as today — user selects it via the dialog); only modes and variable values are added or updated.

The Config page currently fetches the collection's raw tokens but not its themes. It will need to be upgraded to also fetch themes so it can (a) populate the export theme selector and (b) pass the right token data when triggering SD or Figma builds.

**Primary recommendation:** Implement in three tasks — (1) Config page fetch + theme selector UI, (2) theme-aware SD build (merge helper + comment injection), (3) Figma multi-mode export payload.

### Pattern 1: Theme-Aware Token Merge

**What:** Given master collection tokens (as `Record<string,unknown>`) and an optional selected theme (`ITheme | null`), produce a new token JSON where:
- Root groups whose ID maps to `disabled` in `theme.groups` are excluded
- Root groups whose ID maps to `source` contribute their collection-default tokens
- Root groups whose ID maps to `enabled` contribute the theme's `tokens` snapshot for that subtree

**When to use:** Called on the server side (in `/api/build-tokens` or in the Config page client before calling that route) before invoking `buildTokens()`.

**Implementation approach:**

The collection stores tokens as a namespace-wrapped JSON object (e.g. `{ token: { colors: { ... } } }`). Themes store a `TokenGroup[]` snapshot (`theme.tokens`). These are two different shapes — the merge needs to:
1. Convert `theme.tokens` (`TokenGroup[]`) back to the same namespace-wrapped JSON shape using `tokenService.generateStyleDictionaryOutput()`, which already exists in `token.service.ts`
2. Walk the master token JSON by top-level keys; for each key, check if the root group ID matches a group state in `theme.groups`
3. Group IDs in `theme.groups` use slash-separated paths (e.g. `colors`, `colors/brand`) — root groups are single-segment IDs

The cleanest implementation: on the server, re-parse the master tokens into `TokenGroup[]` using `tokenService.processImportedTokens()`, then use group state map to select the right token source per root group, then re-serialize to JSON using `generateStyleDictionaryOutput()`.

**Key insight — group ID matching:** The `ITheme.groups` record keys are the same group IDs as `TokenGroup.id` values (e.g. `colors`, `spacing/base`). Root-level groups have single-segment IDs. The merge only needs to check root-level groups for the state decision (same rule as Phase 11 — child groups are governed by their parent's state).

```typescript
// Source: project codebase — token.service.ts generateStyleDictionaryOutput pattern
// lib/themeTokenMerge.ts (new file)

import type { TokenGroup } from '@/types';
import type { ITheme } from '@/types/theme.types';
import { tokenService } from '@/services/token.service';

/**
 * Merge collection-default tokens with a selected theme.
 * Returns namespace-wrapped token JSON suitable for SD build.
 *
 * @param masterTokens   Raw collection token JSON (namespace-wrapped, e.g. { token: {...} })
 * @param theme          Selected theme, or null for "Collection default"
 * @param namespace      Collection namespace string (e.g. "token")
 * @returns              Merged token JSON
 */
export function mergeThemeTokens(
  masterTokens: Record<string, unknown>,
  theme: ITheme | null,
  namespace: string
): Record<string, unknown> {
  if (!theme) return masterTokens; // Collection default: pass through unchanged

  // Parse master collection into TokenGroup tree
  const { groups: masterGroups } = tokenService.processImportedTokens(masterTokens, namespace);

  // Build a merged TokenGroup[] where:
  // - disabled root groups are removed
  // - source root groups use master group tokens
  // - enabled root groups use theme.tokens snapshot
  const themeGroupMap = new Map<string, TokenGroup>(
    theme.tokens.map(g => [g.id, g])
  );

  const mergedGroups: TokenGroup[] = [];
  for (const masterGroup of masterGroups) {
    const state = theme.groups[masterGroup.id] ?? 'disabled';
    if (state === 'disabled') continue;
    if (state === 'source') {
      mergedGroups.push(masterGroup); // collection default values
    } else {
      // 'enabled': use theme snapshot, fall back to master if not found
      const themeGroup = themeGroupMap.get(masterGroup.id);
      mergedGroups.push(themeGroup ?? masterGroup);
    }
  }

  return tokenService.generateStyleDictionaryOutput(mergedGroups, namespace);
}
```

### Pattern 2: SD Comment Header Injection

**What:** After SD builds each format, prepend a comment to non-JSON formats.

**When to use:** Applied in `buildBrandTokens()` after `formatPlatform()` returns content, or in the API route after `buildTokens()` returns.

The simplest safe approach: inject in the `/api/build-tokens` route handler (or in `BuildTokensPanel` before display) rather than inside `style-dictionary.service.ts` — keeps SD service pure and avoids touching tested code.

The comment format (from CONTEXT.md): `/* Generated by Tokens Manager, theme: [theme name or "Collection default"] */`

```typescript
// In route.ts or service, after buildTokens() returns:
const COMMENT_FORMATS = new Set(['css', 'scss', 'less', 'js', 'ts']);
const headerComment = `/* Generated by Tokens Manager, theme: ${themeLabel} */\n`;

for (const fmt of result.formats) {
  if (!COMMENT_FORMATS.has(fmt.format)) continue;
  for (const output of fmt.outputs) {
    output.content = headerComment + output.content;
  }
}
```

### Pattern 3: Figma Multi-Mode Export Payload

**What:** The current `/api/export/figma` sends a flat `variables` array. The Figma Variables POST API supports `variableModes` and `variableModeValues` arrays to create modes and assign per-mode values in a single request.

**Figma API structure (verified — official docs):**

```typescript
// POST https://api.figma.com/v1/files/:fileKey/variables/local
{
  variableModes: [
    { action: "CREATE", id: "<temp:1>", name: "Default", variableCollectionId: collectionId },
    { action: "CREATE", id: "<temp:2>", name: "Dark",    variableCollectionId: collectionId },
    // ... one per enabled theme
  ],
  variables: [
    {
      action: "CREATE",
      id: "<temp:v1>",
      name: "colors/primary",
      variableCollectionId: collectionId,
      resolvedType: "COLOR",
    },
    // ...
  ],
  variableModeValues: [
    { variableId: "<temp:v1>", modeId: "<temp:1>", value: { r: 1, g: 0, b: 0, a: 1 } },
    { variableId: "<temp:v1>", modeId: "<temp:2>", value: { r: 0, g: 0, b: 1, a: 1 } },
    // ...
  ]
}
```

**Key API facts (verified — Figma official docs):**
- Endpoint: `POST /v1/files/:fileKey/variables/local` (same URL as current implementation)
- Enterprise only — requires full member of Enterprise org with edit access
- Request body size limit: 4 MB
- Mode names max 40 characters; max 40 modes per collection
- Temporary IDs (`<temp:N>` strings) allow referencing newly created objects within the same request
- Response includes `meta.tempIdToRealId` mapping

**Building the multi-mode payload:**

For each token, we need one `variableModeValues` entry per mode (Default + each enabled theme). The token values come from:
- Default mode: master collection tokens
- Theme mode: merge per group state (enabled=theme values, source=master values)

The function `mergeThemeTokens()` can be called once per theme to get the right token values for each mode.

### Pattern 4: Config Page Upgrade

**What:** The current `config/page.tsx` fetches only `tokens`, `namespace`, and `collectionName`. Phase 12 requires it to also fetch themes.

**Current fetch pattern:**

```typescript
const [themes, setThemes] = useState<ITheme[]>([]);

```

The Config page then passes `selectedThemeId` and `themes` down to `BuildTokensPanel`, which calls the `mergeThemeTokens()` helper before posting to `/api/build-tokens`.

### Anti-Patterns to Avoid

- **Modifying `style-dictionary.service.ts` to handle themes:** SD service should remain pure — it accepts token JSON and builds outputs. Theme merging is a caller responsibility. Adding theme logic inside SD service couples concerns.
- **Storing export theme selection in localStorage or DB:** CONTEXT.md explicitly says "not persisted" — keep it as transient `useState` only.
- **Trying to derive group states from the theme for Figma by matching token JSON keys:** Use `theme.groups` record directly to determine group state; it maps group IDs to states.
- **Calling `processImportedTokens` on theme.tokens:** `theme.tokens` is already a `TokenGroup[]`, not raw JSON. Use it directly; do not re-parse it.
- **Sending `variableModes` CREATE for an existing collection that already has modes:** The Figma API supports UPDATE action; however, for simplicity the current implementation creates a new variable collection export each time (same as today). Do not try to diff existing Figma modes — CREATE fresh is safe.

## Common Pitfalls

### Pitfall 1: Theme.tokens vs Raw Collection Tokens are Different Shapes
**What goes wrong:** `theme.tokens` is `TokenGroup[]` (group tree). Collection `tokens` is a raw JSON object (namespace-wrapped). Treating them as the same shape causes incorrect merge behavior.
**Why it happens:** They are stored differently — collection tokens are the SD-format JSON stored in MongoDB; theme tokens are the parsed group tree snapshot.
**How to avoid:** When merging, get master groups via `processImportedTokens(masterTokens, namespace)` and theme groups directly from `theme.tokens`. The `generateStyleDictionaryOutput()` function converts `TokenGroup[]` back to SD JSON.
**Warning signs:** Merge produces empty output or all tokens come from one source.

### Pitfall 2: Group State Applies to Root-Level Groups Only
**What goes wrong:** Trying to match `theme.groups[g.id]` for every group at every level of the tree — including nested children — produces incorrect results because `theme.groups` only maps root-level group IDs.
**Why it happens:** Phase 11 decision: "Root-level source-group guard only (not recursive) — theme.groups maps root-level group IDs; children governed by parent."
**How to avoid:** Only check `theme.groups[g.id]` for root-level groups (those without a parent). Children inherit their parent's state.
**Warning signs:** Source/disabled groups that have nested children show incorrect token values after merge.

### Pitfall 3: Collection Default Token Values for Source Groups
**What goes wrong:** Source groups render theme-overridden values in the export instead of collection-default values.
**Why it happens:** It's tempting to use `theme.tokens` for all groups when a theme is selected. But source groups must use master collection values.
**How to avoid:** For `state === 'source'`, always push the `masterGroup` (from `processImportedTokens`), never the theme group.
**Warning signs:** Source group tokens differ between SD export with theme and export with "Collection default".

### Pitfall 4: Figma API Requires Enterprise — Silent Auth Error
**What goes wrong:** The Figma API returns a 403 or specific error for non-Enterprise accounts. Without surfacing this clearly, users think the export is broken.
**Why it happens:** Figma Variables API is Enterprise-gated (confirmed in official docs). The decision (locked) is to show an inline note + let the user attempt the export + surface the API error in a toast.
**How to avoid:** In `handleExport` in the Config page (or ExportToFigmaDialog equivalent), catch the API error response and show it in a toast with the raw API error message.
**Warning signs:** Users report export button doesn't work with no visible error.

### Pitfall 5: Theme Selector Must Not Affect Figma Export
**What goes wrong:** Passing `selectedThemeId` to the Figma export trigger — Figma export always uses ALL enabled themes.
**Why it happens:** Both export buttons are near the selector; it's easy to pass the same state to both.
**How to avoid:** The Figma export handler always fetches all themes and filters to `enabled-in-at-least-one-group` regardless of which selector value is set. The selector value is only read when triggering SD build.
**Warning signs:** Figma export only sends one mode instead of all enabled themes.

### Pitfall 6: BuildTokensRequest Does Not Currently Accept themeLabel
**What goes wrong:** `BuildTokensRequest` type in `token.types.ts` has no `themeLabel` field. Passing it causes a TypeScript error.
**Why it happens:** The API route performs type checking on the request body.
**How to avoid:** Extend `BuildTokensRequest` to include `themeLabel?: string` before wiring comment injection. The comment injection logic then reads `body.themeLabel ?? 'Collection default'`.

```typescript
const payload = {
  variableModes: [
    { action: "CREATE", id: "temp:mode:default", name: "Default", variableCollectionId: collectionId },
    ...enabledThemes.map((theme, i) => ({
```

## Open Questions

1. **How to handle themes where no groups are enabled (all groups disabled or source)**
   - What we know: CONTEXT.md says "No enabled themes: export Default mode only (graceful fallback, no error)" — but this is about Figma specifically when zero themes have any enabled groups.
   - What's unclear: For SD export — if the user picks a theme where all groups are `source` or `disabled`, the merge produces only source-group tokens. This is valid — it uses collection-default values for source groups and excludes disabled groups.
   - Recommendation: No special handling needed for SD. For Figma, if a theme has zero enabled groups (all disabled or all source), it still becomes a mode — its values are all collection-default (source groups) or the variable is absent (disabled groups).

2. **Where to call `mergeThemeTokens` — client-side (Config page) or server-side (build-tokens route)**
   - What we know: `tokenService.generateStyleDictionaryOutput()` and `processImportedTokens()` are available both client and server. The collection tokens are already loaded in the Config page's client state. Adding merge logic client-side avoids a round-trip.
   - What's unclear: Large collections might make the client-side merge slow; but for typical collections this is negligible.
   - Recommendation: Call `mergeThemeTokens` client-side in the Config page before the POST, same pattern as `BuildTokensPanel` currently sends the raw tokens. This keeps the API route simple and avoids adding theme fetching to the server route.

3. **Figma export — whether to CREATE or UPDATE existing variables**
   - What we know: Current implementation CREATEs variables from scratch each time. The Figma API supports UPDATE action with the variable's real ID. Re-creating on each export is idempotent from the user's perspective.
   - What's unclear: If the same collection in Figma already has variables, CREATE will duplicate them.
   - Recommendation: Keep the existing CREATE approach for now (same as before). The ExportToFigmaDialog already shows a collection picker so the user chooses where to export. Adding UPDATE/diff logic is THEME-06+ territory.
